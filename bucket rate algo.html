How the Token Bucket Algorithm Works

1

Initial State & Capacity

The **token bucket** starts with a maximum capacity (e.g., 5 tokens). Think of it like a fixed-size container that can hold a certain number of tokens. Initially, it's often full.

2

Token Refill Rate

Tokens are **automatically added** to the bucket at a constant, predefined rate (e.g., 1 token every 2 seconds). This ensures a steady supply of "permission slips" for requests. The bucket will never exceed its maximum capacity.

3

Request Arrival & Consumption

When an **API request arrives**, the system checks if there are enough tokens in the bucket. Typically, each request consumes one token. If tokens are available, one is consumed, and the request is processed.

4

Handling Bursts (Burst Capacity)

The bucket's capacity allows for **bursts of requests**. If the bucket is full, multiple requests can be served instantly until the tokens are depleted. This is a key advantage, accommodating sudden spikes in traffic.

5

Rate Limiting Excess Requests

If a request arrives and the **bucket is empty**, it means the rate limit has been exceeded. The request is then **denied** (e.g., with a "429 Too Many Requests" error) until more tokens become available through the refill process.

6

Continuous Operation

The refill and consumption process is **continuous**. Tokens are constantly refilling, and requests are constantly consuming them, ensuring that the API operates within its defined limits while allowing for some flexibility.
